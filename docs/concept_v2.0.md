# 0. Исходная мотивация

Цель проекта — не ускорить нейросеть как таковую и не заменить оптимизатор.

Цель — изменить пространство поиска так, чтобы:

* его можно было исследовать адаптивно,
* вычислительный ресурс тратился только там, где это даёт информацию,
* при этом сохранялись взаимосвязи признаков (не «ломались фичи»).

Ключевая идея:

> Размерность — это не фиксированное число, а глубина уточнения. Уточнение должно происходить только там, где это оправдано информационно и бюджетно.

# 1. Базовая модель

## 1.1 Пространство

Есть пространство состояний X (любой природы: признаки, латент, активации, данные).

Есть функция информативности ρ(x).

ρ(x) — мера того, где уточнение имеет смысл.

## 1.2 Уточнение

Процесс:

1. Стартуем с грубого приближения (coarse уровень).
2. Вычисляем ρ.
3. Решаем, стоит ли делить регион (split).
4. Уточняем только выбранные регионы.
5. Повторяем до остановки.

Уточнение всегда имеет форму:

```
output = coarse + delta
```

где delta:

* инициализируется нулём,
* ограничена по энергии,
* при отключении любого текущего уровня возвращает валидное представление предыдущего уровня,
* может ухудшать локальную согласованность у границ.

Halo применяется на каждом уровне уточнения (с учётом правила применимости по топологии, см. раздел 6).

# 2. Деревья и «кусты»

## 2.1 Дерево

Дерево — журнал маршрутов уточнения.

Каждый путь от корня к листу — последовательность решений split.

Структура должна быть GPU-дружественной (плоская упаковка, без pointer chasing).

**Принцип адресации (v1.8.1):** адресация должна быть на уровне абстракции операции. Refinement оперирует тайлами — адресация tile-level, не element-level. Внутри тайла данные плотные (dense intra-tile). Снаружи — sparse по тайлам. Глобальный element-level reverse_map запрещён (exp10: VRAM +38.6%, конструктивный провал). Lookup — только на active/support set O(k), не на всё пространство O(M).

**Layout policy по типам пространств (v1.8.2, exp10e-j, FINAL):**
- scalar_grid → D_direct (packed tiles + tile_map) — production layout (exp10g: both contours PASS)
- vector_grid → D_direct — production layout (exp10h: 72/72 PASS both contours)
- tree_hierarchy → Hybrid per-level: D_direct where p_l < 0.40 AND operator is compute-heavy (matmul-like); A_bitset elsewhere. Upper levels (small N_l, high occupancy) → A_bitset. Lower levels (large N_l, low occupancy, heavy compute) → D_direct. Stencil ops: D saves memory but NEVER wins time → A_bitset. (exp10j: 158K trials, break-even stable across all branching factors)
- irregular_graph / spatial → D_blocked conditional (spatial partition, cbr ≤ 0.35) (exp10i)
- irregular_graph / scale-free → blocked layout rejected (cbr=0.66); A_bitset fallback (exp10i)

Для графов: fixed-size blocks НЕ универсальная абстракция. Compute-path здоровый (Contour B 100%), представление данных больное (padding 50-97%). Графы расщепились на два класса по наличию пространственной структуры.

**Layout Selection Invariant (v1.8.3) — ГИПОТЕЗА, не доказана**

Идея: таблица выше — предвычисленный кэш функции C(I, M, p), где layout = argmin C. Три оси:

**Ось 1 — Топологическая изотропия I(X).** Энтропия распределения степеней H(D). Описывает неравенство степеней, но не геометрию связей. Ограничение: random 4-regular graph имеет I=0 (как grid), но 100% cache miss rate. I слепа к пространственной запутанности.

**Ось 2 — Метрический разрыв M(X, L).** Связана с алгебраической связностью (спектральная щель λ₂ Лапласиана графа). Вектор Фидлера (собственный вектор λ₂) — теоретически оптимальная 1D проекция графа в линейную память. Ограничение: λ₂ — глобальная метрика; два идеальных кластера + один мост → λ₂ ≈ 0, но GPU считает внутри кластеров с 99.9% cache hit rate. Глобальный скальпель не видит локальное здоровье.

**Ось 3 — Динамическая плотность p.** Occupancy k/N. Порог p* ≈ 0.40 стабилен (exp10j). Единственная ось, подтверждённая экспериментально.

**Экспериментальная проверка (exp10k, 810 trials):**
- Sparse vs dense: **подтверждено** — A_bitset = 0 побед из 810. Sparse всегда лучше.
- D_direct vs D_blocked: **не разрешимо** — boundary smoothness 0.496 (jagged). Поверхность C(I, M, p) не гладкая.

**Диагноз:** метрики I и M вычислены на полном пространстве X, а не на активном подграфе X_active. Эффективная топология для GPU — это X_active, у которого другая энтропия и другой метрический разрыв. Необходима коррекция: I_active = H(D|X_active), M_local = средний cache miss rate на рёбрах X_active. См. `experiments/exp_deferred_revisit/deferred_revisit_note.md`, раздел 7.

**Текущий статус:** гипотеза отложена (Track C). Таблица policy работает как эмпирическая классификация и не требует непрерывного закона. При появлении нового типа пространства — revisit с исправленными метриками.

Runtime-профилировщик НЕ предусмотрен: в текущем scope типы пространств известны статически.

**Альтернативная интерпретация (теоретическая, не проверена экспериментально):**
Дерево может рассматриваться как траектория RG-потока (renormalization group flow). Узлы — состояния системы при разных масштабах, рёбра — операции coarse-graining. Fixed points потока — домены, где дальнейший refinement перестаёт менять эффективное описание. Эта интерпретация пока не влияет на архитектуру, но мотивирует Scale-Consistency Invariant (раздел 8).

## 2.2 Куст

Куст = множество путей, ведущих к одному смыслу.

Для первого прототипа используется метрика расстояния в дереве (через LCA / общий префикс).

# 3. Функция информативности ρ(x)

Семантика дерева определяется ρ.

## 3.1 Компоненты ρ

ρ может включать:

* Residual — ошибку текущей аппроксимации (основной сигнал).
* HF energy — структурную энергию (Лапласиан, градиент).
* Variance — локальную дисперсию / disagreement.
* Окупаемость — ожидаемый gain vs. стоимость.
* Teacher-guided сигнал.
* D_parent — межмасштабный drift (см. раздел 8). Контекстный сигнал, не самодостаточный.

ρ может быть иерархическим (дешёвый критерий → дорогой критерий).

## 3.2 Комбинирование сигналов (Exp0.4–Exp0.7b)

Один сигнал = одна предвзятость. Residual слеп к скрытой структуре при шуме; HF ловит ложные частоты; Variance любит шум.

Архитектура комбинирования — двухстадийный гейт:

**Stage 1** — «Residual здоров?» (бинарная проверка):

* Если residual стабилен (instability < порог, FSR < порог) → веса коллапсируют в residual-only.
* Если нет → переход в Stage 2.

**Stage 2** — utility-based soft weights:

* Каждый эксперт (resid, var, hf) получает вес по полезности: U_i = median(gain) − λ·FSR − μ·instability.
* Residual имеет минимальный гарантированный вес (prior).
* Веса сглажены EMA с гистерезисом.

Нормализация — квантильная (ранговая), не абсолютная.

**Экспериментальный статус:**

* Exp0.4: Residual-only = оракул на чистых данных.
* Exp0.5: Оракул ломается при шуме (corr 0.90 → 0.54), деградированном coarse.
* Exp0.6: Бинарный переключатель resid/combo работает, но alias — пограничный случай.
* Exp0.7: Soft gating побеждает на noise/spatvar/jpeg (+0.08–1.10 дБ), но проигрывает на clean/blur (−1.6–2.5 дБ) из-за «демократичности» softmax.
* Exp0.7b: Двухстадийный гейт решает проблему clean/blur (Δ ≈ 0) и сохраняет выигрыш на noise/spatvar (+0.77–1.49 дБ). JPEG — единственный минус (−0.21 дБ, вопрос тонкой настройки порога instability).

**Вывод:** двухстадийный гейт — правильная архитектура. Residual правит, пока достоин; когда ломается — плавно уступает комбинации.

## 3.3 Трёхслойная декомпозиция ρ (exp17, 22–23 марта 2026)

Phase 3 показала: монолитная ρ смешивает три ортогональных сигнала, что делает дерево refinement непереиспользуемым. Решение — каскадная декомпозиция:

**Layer 0 (Topology)** — "как устроено пространство?" Data-independent. Вычисляется один раз при инициализации. Для graph: Leiden + curvature + PageRank + boundary anomalies → per-unit structural score + cluster_ids. Для grid: spatial quadrant blocks (тривиальная топология). Для tree: depth-band grouping.

**Layer 1 (Presence)** — "где есть данные?" Data-dependent, query-independent. Variance GT per unit → presence score. Порог: **каскадные квоты** (Variant C) — каждый L0-кластер гарантирует min(1, ceil(cluster_size × budget_fraction)) выживших. Топология диктует квоты, ни один регион не вымирает.

**Layer 2 (Query)** — "где именно нужно уточнение?" Task-specific. Три сменных query-функции на одном frozen tree: MSE, max absolute error, HF residual. Дёшево — дерево уже есть.

**Frozen Tree** — снапшот L0+L1. Переиспользуемый пространственный индекс. Аналог R-tree в пространственных БД.

**Streaming Pipeline** — покластерная обработка (L0→L1→L2 per cluster) с L0-priority ordering и глобальным budget cap. Первые результаты появляются после обработки первого кластера.

**Экспериментальный статус (exp17):** 1080 конфигов, 4 spaces × 3 scales (100/1K/10K) × 8 approaches × 20 seeds. Reusability: 12/12 PASS (min ratio 0.838). PSNR: 2-4 dB ниже single_pass на grid (цена pruning), паритет на graph/tree. Streaming 10-20% быстрее batch. kdtree (scipy C) быстрее на single query, three_layer выигрывает при ≥2 запросах на tree_hierarchy. Roadmap: C/Cython оптимизация scoring даст мультипликативный эффект.

**Связь с двухстадийным гейтом (3.2):** гейт остаётся внутри Layer 2 — он определяет какой query signal использовать (residual-only или combo). Layers 0-1 работают до гейта и не зависят от него.

# 4. Политика интересности

Баланс сигналов зависит от глубины L.

## 4.1 Абсолютная мощность

На coarse уровнях приоритет у крупных структур.

Вес абсолютной мощности затухает с глубиной.

## 4.2 Локальная аномальность

Чувствительность к локальной дисперсии усиливается с глубиной.

## 4.3 Окупаемость

Split допускается только если выигрыш перекрывает стоимость.

## 4.4 Бюджет

Жёсткие ограничения: max_depth, max_nodes, max_cost, max_boundary_budget.

## 4.5 Детектор мусора

Если средний выигрыш за K сплитов < δ (5–10% от предыдущего уровня) — дробление шума подавляется.

# 5. Schedule: управление строгостью (Exp0.8)

## 5.1 Governor (EMA-контроллер) — обязателен

**Объект управления:** strictness — квантильный порог отбора кандидатов на refine.

**Сигнал обратной связи:** EMA фактического cost/step (после warmup).

**Цель:** удерживать cost/step в коридоре вокруг target_per_step.

**Предохранители:**

* Δstrictness за шаг ≤ clamp (anti-oscillation).
* hard_cap_per_step = 3× target (safety fuse).
* Warmup: первые N шагов strictness не двигается.
* Compliance асимметричная: overbudget penalty > underbudget penalty.

**Экспериментальный результат (Exp0.8v5):**

* StdCost −50%, P95 с 11.0 до ~6.5, compliance penalty с ~3.2 до ~0.5.
* PSNR чуть ниже (−0.24 дБ clean, −0.68 дБ shift).
* **Вывод:** governor нужен не для качества, а для контроля бюджета. Без него бюджет — декларация.

## 5.2 Условие применимости governor

Если при max strictness n_candidates ≥ hard_cap — контроллер бесполезен. Лечится параметризацией задачи.

# 6. Boundary-aware refinement (обязательное требование)

## 6.1 Правило применимости Halo по топологии

Halo применяется **не безусловно**, а при выполнении двух условий:

a) **Boundary parallelism ≥ 3** — минимум 3 независимых cross-edge на границе тайла (min_cut ≥ 3).
b) **No context leakage** — расширение halo не затекает в несвязанные тайлы.

Оба условия должны выполняться одновременно.

### По топологии:

| Топология | Halo | Обоснование |
|---|---|---|
| Grid (tile_size ≥ 3) | **ВСЕГДА** применять | Boundary parallelism гарантирован |
| k-NN граф с пространственной кластеризацией | **Применять** | min_cut >> 3 |
| Tree / Forest | **НИКОГДА** | min_cut=1, context leakage гарантирован |
| DAG | **Проверять per-case** | Зависит от конкретной структуры |

### Корневая причина отказа на деревьях

Деревья имеют single-edge bottleneck (min_cut=1 между любыми двумя поддеревьями), sibling bleed (halo одного потомка затекает в поддерево соседнего потомка) и экстремальную асимметрию S/V (surface/volume ratio). Эти три фактора делают halo на деревьях не просто бесполезным, а вредным.

## 6.2 Минимальное правило (для применимых топологий)

* overlap ≥ 3–4 элемента при tile_size ≥ 16.
* Blending (cosine feather) выполняется относительно coarse.

**Важно:** halo отвечает за локальную геометрическую корректность на границах тайлов. Это ортогонально Scale-Consistency Invariant (раздел 8), который отвечает за межмасштабную семантическую корректность. Одно не заменяет другое.

# 7. Probe-budget exploration (обязательное требование)

Каждый шаг:

* 90–95% бюджета → exploitation.
* 5–10% бюджета → probe.

**Уточнённая роль probe (v1.6):** probe — не просто exploration, а механизм защиты от ложных fixed points. Если refinement сходится к состоянию "ничего интересного нет", probe периодически проверяет это утверждение. По аналогии с тепловыми флуктуациями в физике: случайные возмущения, предотвращающие вырождение в локальный минимум.

Overlap не заменяет exploration. Exploration не лечит швы. Оба механизма обязательны.

# 8. Scale-Consistency Invariant (новый раздел, v1.6; обновлён v1.7)

## 8.1 Мотивация

Открытый вопрос v1.5 "как не сломать фичи" формализуется через требование масштабной согласованности представлений.

Halo обеспечивает локальную корректность на границах (геометрический уровень). Этого недостаточно: refined уровень может быть гладким по швам, но при этом семантически противоречить coarse уровню на своём масштабе.

## 8.2 Асимметричный инвариант

Ключевой принцип: **coarse — якорь, delta — подчинённая поправка**.

`refined = coarse + delta`

Направление одностороннее: refined не переопределяет coarse, а живёт внутри его допуска. Строгое равенство `coarse = R(refined)` — ловушка: оно разрушает матрёшку и превращает дерево в пирамиду зависимостей снизу вверх.

**Формальное требование:**

```
‖R(delta)‖ / (‖delta‖ + ε) < τ_rel
```

Смысл: delta должна быть невидима с уровня выше. Refinement может добавлять детали, но не должен проталкивать наверх новый coarse-смысл контрабандой.

## 8.3 Оператор coarse-graining

Для работы инварианта необходимо зафиксировать согласованную пару **(R, Up)**:

* **R** — оператор укрупнения: `low-pass filter + decimation`. Выбор: gaussian blur σ=3.0 + decimation (линейный, GPU-дешёвый, согласуется с coarse + delta). **Обновлено v1.7:** σ увеличен с 1.0 до 3.0 — σ=1.0 был недостаточно агрессивен как LP-фильтр, positive deltas сохраняли энергию после R. **Known limitation:** σ=3.0 выбрано как наименьшее целое значение, прошедшее kill criteria в грубом sweep [0.5, 1.0, 2.0, 3.0]. Fine-grained sweep не проводился; оптимальное σ может зависеть от tile_size и типа пространства. Запланирован отдельный эксперимент (SC-σ sweep).
* **Up** — оператор восстановления LF-компоненты в пространство delta: `bilinear upsampling`. Это **не** обратный оператор к R (он не существует), а проекция coarse-компоненты обратно в исходный масштаб.

Пара (R, Up) фиксируется до первого эксперимента. Разные пары дают разные физики дерева и разные fixed points. Выбор — архитектурный, не технический.

Требование идемпотентности: `‖R(coarse) − coarse_downsampled‖ ≈ 0`, где coarse_downsampled — coarse в масштабе R. Если не выполнено — дерево разваливается. (R включает decimation, поэтому сравнение ведётся в пространстве меньшего масштаба, а не с исходным coarse.)

## 8.4 Метрики

Две неэквивалентные диагностические оси (разные числители, но обе строятся через пару (R, Up) — статистическая независимость не гарантирована):

**D_parent** — утечка delta в родительский масштаб (обновлено v1.7):
```
D_parent = ‖R(delta)‖ / (‖delta‖ + ε)
```

**Обновление v1.7:** старая формула `D_parent = ‖R(delta)‖ / (α·‖coarse‖ + β)` заменена. Причина: знаменатель `α·‖coarse‖ + β` одинаков для всех delta на одном тайле и не несёт дискриминативного сигнала. Новая интерпретация: «какая доля энергии delta является низкочастотной?» (lf_frac) — операционально корректная мера.

**D_hf** — высокочастотная чистота самой delta:
```
delta_HF = delta - Up(R(delta))   # HF-остаток после удаления LF-компоненты
D_hf = ‖delta_HF‖ / (‖delta‖ + ε)
```

Интерпретация комбинаций:

| D_parent | D_hf | Смысл |
|---|---|---|
| высокий | низкий | delta в основном LF и лезет в родителя. Нарушение инварианта. |
| низкий | высокий | delta detail-like, родителя не трогает. Норма. |
| оба средние | — | Пограничная зона; смотреть residual/gain/probe. |

### Результаты кросс-пространственной валидации (v1.7)

| Пространство | AUC D_parent |
|---|---|
| T1 scalar grid | 1.000 |
| T2 vector grid | 1.000 |
| T3 irregular graph | 1.000 |
| T4 tree hierarchy | 0.824 |

Пространственно-специфическая настройка не требуется. Формула D_parent (lf_frac) работает из коробки на всех 4 типах пространств.

## 8.5 Порог τ_rel / τ_parent

τ_rel — концептуальный относительный порог из формулы инварианта. В операциональной реализации он материализуется как **τ_parent** — data-driven порог, возможно зависящий от уровня L:

```
τ_rel  →  τ_parent[L]
```

Порог не константа: относительная нормировка встроена в формулу, но на fine уровнях delta должна становиться всё более HF-чистой, поэтому τ_parent[L] может убывать с глубиной. Это выясняется из baseline-эксперимента, а не задаётся заранее.

ε в знаменателе — стабилизатор от деления на ноль на почти пустых регионах.

**τ_parent устанавливается data-driven** по baseline-эксперименту (см. раздел 8.6 и `scale_consistency_verification_protocol_v1_0.md`), а не вручную.

## 8.6 Baseline-эксперимент (обязательный первый шаг)

До введения enforcement необходимо собрать распределения D_parent и D_hf на:

* **Positive baseline** — заведомо корректные refinement-случаи (чистые синтетические данные, near-GT delta).
* **Negative baseline** — заведомо плохие случаи (искусственный LF drift, намеренный coarse shift, семантически неправильная delta).

Мерить по срезам: глобально, по уровням дерева, по типам структуры (гладкое, граница, текстура, шум).

**Kill criterion (фиксируется до запуска baseline):**

Если separability между positive и negative недостаточна (по AUC / effect size / quantile separation) — **пересматриваются метрики или пара (R, Up)**, а не подгоняются пороги τ. Плохая separability означает, что сигнал не несёт нужной дискриминативной информации, а не что порог выбран неудачно.

**Статус (v1.7): SC-baseline ВЫПОЛНЕН и ПРОЙДЕН.**

* D_hf: AUC=0.806, d=1.34 (pass)
* D_parent (с исправлениями формулы и R): AUC в диапазоне [0.824, 1.000] на 4 пространствах (pass)
* Генератор coarse_shift исправлен (пространственно-когерентные знаковые поля)
* Статус: baseline завершён, готовность к SC-enforce

## 8.7 Enforcement

После baseline-валидации:

```python
if D_parent > τ_parent:
    damp delta
    reject split
    increase strictness locally
```

D_parent используется также как контекстный сигнал в ρ:
* высокий D_parent + высокий residual + стабильный gain → регион структурно интересен → повысить приоритет probe
* высокий D_parent + низкий gain + нестабильная delta → проблема в механике refinement → не поощрять split

## 8.8 Scale-stable fixed point

Локальная остановка refinement определяется не только бюджетом, но и достижением scale-stable fixed point.

Узел считается scale-stable, если одновременно:
1. `gain_from_split < τ_gain`
2. `D_parent < τ_parent`
3. Это держится устойчиво K шагов подряд.

Смысл: refinement уже не добавляет ничего значимого ни локально, ни на масштабе выше.

**Probe остаётся обязательным** как страховка от ложной стабилизации (ложный fixed point неотличим от настоящего без внешней проверки).

Бюджетный governor остаётся как глобальный предохранитель. Локальная остановка по fixed point и глобальный бюджет — ортогональные механизмы.

# 8A. Детерминизм и воспроизводимость (новый раздел, v2.0)

## 8A.1 Мотивация

В v1.3 существовал инвариант «дерево строится по стабильному правилу». К v1.7 он был утерян — заменён другими инвариантами без явного наследования.

Без формализованного детерминизма:
* **Тестируемость** (Track A): невозможно ловить регрессии, если «правильное» дерево каждый раз выглядит иначе. Изменение в коде неотличимо от статистического шума.
* **Воспроизводимость** (Track B): результаты, которые нельзя повторить при фиксированных условиях, не подлежат научному анализу.
* **Content-addressable cache**: хэширование маршрутов теряет эффективность при нестабильных путях (кэш контента не страдает, но переиспользование маршрутов — да).

## 8A.2 Два уровня требований

### Уровень 1: Детерминизм при фиксированном seed (Hard Constraint)

**Инвариант:** одинаковые данные + ρ + seed + бюджет = одинаковое дерево (побитовое совпадение).

Три обязательных компонента:

1. **Канонический порядок обхода (tie-break).** При равном ρ порядок split-решений определяется каноническим индексом тайла (Z-order / Morton). Кто по индексу меньше — тот первый. Никакой зависимости от порядка итерации в хэш-таблице или гонки потоков.

2. **Детерминированный probe.** Probe seed = f(координаты_тайла, уровень_глубины, global_seed). Pseudo-random, но воспроизводимый. «Разведка» предсказуема при фиксированном seed.

3. **Изоляция governor'а от порядка обработки.** Накопление EMA и решения governor'а не зависят от порядка обработки соседних веток. Обработка — в каноническом порядке, EMA-апдейт — строго после полного шага.

**Enforcement:** Unit-тесты: два прогона с одинаковыми входами → побитовое совпадение дерева. Любое нарушение — fail.

### Уровень 2: Статистическая устойчивость по seeds (Soft Constraint)

**Инвариант:** при разных seeds метрики (PSNR, cost, compliance, SeamScore) статистически устойчивы.

Формально: для N ≥ 10 seeds коэффициент вариации CV(metric) < τ_cv (τ_cv устанавливается data-driven по baseline).

Разные seeds дают разные деревья — это нормально. Но метрики должны быть стабильны. Если одна ρ при seed=42 даёт PSNR 30 дБ, а при seed=43 даёт 25 дБ — система нестабильна и выводы по ней невалидны.

**Уже частично на месте:** 10–20 seeds в протоколах экспериментов, Holm-Bonferroni коррекции. Формализация требует только записи как инварианта и определения τ_cv.

## 8A.3 Связь с другими механизмами

| Механизм | Источник недетерминизма | Решение |
|---|---|---|
| Probe | Стохастический выбор тайлов | seed = f(coords, level, global_seed) |
| Governor (EMA) | Порядок обработки влияет на накопление | Канонический порядок + апдейт после полного шага |
| Split при равном ρ | Зависимость от реализации обхода | Tie-break по Z-order/Morton индексу |
| Halo overlap на GPU | Overlapping writes при параллельном accumulation | Exp0.9h (запланирован) |
| Двухстадийный гейт | Порядок оценки экспертов | Фиксированный порядок, без зависимости от данных |

## 8A.4 Экспериментальная валидация

Два эксперимента (см. experiment_hierarchy.md):

1. **DET-1: Seed determinism.** Два прогона, идентичные входы, один seed → побитовое совпадение дерева. CPU и GPU отдельно. Kill criterion: любое расхождение = fail.

2. **DET-2: Cross-seed stability.** N=20 seeds, 4 пространства, 2 бюджета. Мерить CV для PSNR, cost, compliance, SeamScore. Kill criterion: CV > 0.10 для любой метрики = fail (порог предварительный, уточняется по baseline).

DET-1 — предусловие Track A (Instrument Readiness Gate, stability pass). DET-2 — предусловие Track B.

# 11. Ключевые инварианты

1. Уточнение аддитивно (residual).
2. Halo обязателен (локальная граница тайлов) — с учётом правила применимости по топологии (раздел 6).
3. Exploration обязателен (probe как защита от ложных fixed points).
4. Стоимость управляется governor'ом (EMA-контроллер) в рамках бюджета.
5. Система не создаёт искусственные швы.
6. Система не становится структурно слепой.
7. ρ определяет семантику карты; комбинирование через двухстадийный гейт.
8. **Scale-consistency:** delta не переопределяет семантику родительского масштаба. `‖R(delta)‖ / (‖delta‖ + ε) < τ_rel`. Пара (R, Up) зафиксирована. Пороги data-driven.
9. **Кросс-пространственная валидация:** любое утверждение об «произвольных пространствах» проверяется на ≥4 типах пространств (scalar grid, vector grid, irregular graph, tree hierarchy). Результат на одном типе НЕ достаточен.
10. **Детерминизм при фиксированном seed:** одинаковые данные + ρ + seed + бюджет = одинаковое дерево. Канонический порядок обхода (tie-break по Z-order), детерминированный probe (seed от координат), изоляция governor от порядка обработки. (Раздел 8A.)
11. **Статистическая устойчивость по seeds:** метрики стабильны при разных seeds (CV < τ_cv). Разные деревья допустимы, разные выводы — нет. (Раздел 8A.)
12. **Layout–topology correspondence:** формат укладки памяти определяется вектором свойств пространства [I(X), M(X,L), p], а не типовым именем. I = энтропия степеней (изотропия), M = ранговая корреляция Кендалла между топологической и физической дистанцией (метрический разрыв), p = occupancy (динамическая плотность). Текущая таблица policy — предвычисленный кэш для известных классов. (Раздел 2.1.)

# 12. Главный принцип Curiosity

Curiosity — система, которая:

1. Считает только там, где есть сигнал.
2. Делает это без создания искусственных швов (halo).
3. Не становится слепой к внутренней структуре (probe / защита от ложных fixed points).
4. Управляет бюджетом осознанно (governor), а не декларативно.
5. Адаптирует функцию интересности к условиям (двухстадийный гейт), а не полагается на один датчик.
6. **Не разрушает семантику родительского масштаба при уточнении (scale-consistency).**
7. **Воспроизводима при фиксированных условиях и статистически устойчива при варьировании seed.**
8. **Выбирает формат памяти по свойствам пространства (layout–topology correspondence), а не по имени типа.**

Четыре закона сохранения:
* budget conservation → budget governor
* scale consistency → D_parent constraint + enforcement
* process determinism → canonical order + deterministic probe + governor isolation
* layout–topology correspondence → layout = f(I, M, p); таблица policy = предвычисленный кэш

Формально: refinement must be boundary-aware, must include controlled exploration, must be budget-governed, must preserve scale-consistent representation, must be deterministic given fixed inputs, and must select memory layout by topological properties of the space.

## 14. Enox-инфраструктура: паттерны наблюдения (v1.8.4, 21 марта 2026)

Четыре инфраструктурных паттерна, заимствованных из Enox framework (идеи, не код). Все — чистое наблюдение/аннотация: **никогда не модифицируют pipeline state**. Все дефолты = off.

### 14.1 RegionURI

Детерминированный адрес юнита: `SHA256(parent_id | op_type | child_idx)`, усечённый до 16 hex символов. Связывает юнит через весь pipeline: init → dedup → gate → enforce → final. Аналог Module A (каноникализация) для внутренних объектов pipeline.

### 14.2 DecisionJournal

Append-only лог каждого решения: region_id, tick, gate_stage (stage1_healthy / stage2_utility / enforce), decision (pass / damped / rejected / d2_skip), metrics_snapshot, thresholds_used. No-op при отключении. Превращает pipeline из чёрного ящика в аудируемый процесс.

### 14.3 MultiStageDedup

Трёхуровневая проверка перед refinement: (a) exact hash — состояние не изменилось, (b) metric distance — |Δρ| < ε, (c) external policy. В текущем single-pass режиме (ε=0.0) ни один уровень не срабатывает — это **заготовка** для multi-pass итеративного уточнения, где повторная обработка неизменившегося юнита — чистая трата бюджета.

### 14.4 PostStepSweep

После refinement loop: обход дерева, поиск sibling'ов с одинаковым dirty-signature. Если > 5% пар идентичны — сигнал для будущего merge. Работает только на tree_hierarchy (у сеток и графов нет parent-child структуры).

### 14.5 Принцип интеграции

Инфраструктура наблюдения дешевле всего встраивается в простой pipeline. В multi-pass режиме (adaptive budgets) та же интеграция потребовала бы рефакторинга уже сложного кода. Поэтому паттерны закладываются сейчас, активируются позже.
