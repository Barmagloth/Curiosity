# Curiosity — Концептуальная фиксация (v1.5)

Документ фиксирует текущую логику проекта так, чтобы через год можно было восстановить ход мысли без внешнего контекста.

Обновлено после экспериментов Exp0.1–Exp0.8.

---

# 0. Исходная мотивация

Цель проекта — не ускорить нейросеть как таковую и не заменить оптимизатор.

Цель — изменить пространство поиска так, чтобы:

* его можно было исследовать адаптивно,
* вычислительный ресурс тратился только там, где это даёт информацию,
* при этом сохранялись взаимосвязи признаков (не «ломались фичи»).

Ключевая идея:

> Размерность — это не фиксированное число, а глубина уточнения. Уточнение должно происходить только там, где это оправдано информационно и бюджетно.

---

# 1. Базовая модель

## 1.1 Пространство

Есть пространство состояний X (любой природы: признаки, латент, активации, данные).

Есть функция информативности ρ(x).

ρ(x) — мера того, где уточнение имеет смысл.

## 1.2 Уточнение

Процесс:

1. Стартуем с грубого приближения (coarse уровень).
2. Вычисляем ρ.
3. Решаем, стоит ли делить регион (split).
4. Уточняем только выбранные регионы.
5. Повторяем до остановки.

Уточнение всегда имеет форму:

```
output = coarse + delta
```

где delta:

* инициализируется нулём,
* ограничена по энергии,
* при отключении любого текущего уровня возвращает валидное представление предыдущего уровня,
* может ухудшать локальную согласованность у границ.

Halo применяется на каждом уровне уточнения.

---

# 2. Деревья и «кусты»

## 2.1 Дерево

Дерево — журнал маршрутов уточнения.

Каждый путь от корня к листу — последовательность решений split.

Структура должна быть GPU-дружественной (плоская упаковка, без pointer chasing).

## 2.2 Куст

Куст = множество путей, ведущих к одному смыслу.

Для первого прототипа используется метрика расстояния в дереве (через LCA / общий префикс).

---

# 3. Функция информативности ρ(x)

Семантика дерева определяется ρ.

## 3.1 Компоненты ρ

ρ может включать:

* Residual — ошибку текущей аппроксимации (основной сигнал).
* HF energy — структурную энергию (Лапласиан, градиент).
* Variance — локальную дисперсию / disagreement.
* Окупаемость — ожидаемый gain vs. стоимость.
* Teacher-guided сигнал.

ρ может быть иерархическим (дешёвый критерий → дорогой критерий).

## 3.2 Комбинирование сигналов (Exp0.4–Exp0.7b)

Один сигнал = одна предвзятость. Residual слеп к скрытой структуре при шуме; HF ловит ложные частоты; Variance любит шум.

Архитектура комбинирования — двухстадийный гейт:

**Stage 1** — «Residual здоров?» (бинарная проверка):

* Если residual стабилен (instability < порог, FSR < порог) → веса коллапсируют в residual-only.
* Если нет → переход в Stage 2.

**Stage 2** — utility-based soft weights:

* Каждый эксперт (resid, var, hf) получает вес по полезности: U_i = median(gain) − λ·FSR − μ·instability.
* Residual имеет минимальный гарантированный вес (prior).
* Веса сглажены EMA с гистерезисом.

Нормализация — квантильная (ранговая), не абсолютная.

**Экспериментальный статус:**

* Exp0.4: Residual-only = оракул на чистых данных.
* Exp0.5: Оракул ломается при шуме (corr 0.90 → 0.54), деградированном coarse.
* Exp0.6: Бинарный переключатель resid/combo работает, но alias — пограничный случай.
* Exp0.7: Soft gating побеждает на noise/spatvar/jpeg (+0.08–1.10 дБ), но проигрывает на clean/blur (−1.6–2.5 дБ) из-за «демократичности» softmax.
* Exp0.7b: Двухстадийный гейт решает проблему clean/blur (Δ ≈ 0) и сохраняет выигрыш на noise/spatvar (+0.77–1.49 дБ). JPEG — единственный минус (−0.21 дБ, вопрос тонкой настройки порога instability).

**Вывод:** двухстадийный гейт — правильная архитектура. Residual правит, пока достоин; когда ломается — плавно уступает комбинации.

# 4. Политика интересности

Баланс сигналов зависит от глубины L.

## 4.1 Абсолютная мощность

На coarse уровнях приоритет у крупных структур.

Вес абсолютной мощности затухает с глубиной.

## 4.2 Локальная аномальность

Чувствительность к локальной дисперсии усиливается с глубиной.

## 4.3 Окупаемость

Split допускается только если выигрыш перекрывает стоимость.

## 4.4 Бюджет

Жёсткие ограничения: max_depth, max_nodes, max_cost, max_boundary_budget.

## 4.5 Детектор мусора

Если средний выигрыш за K сплитов < δ (5–10% от предыдущего уровня) — дробление шума подавляется.

# 5. Schedule: управление строгостью (Exp0.8)

## 5.1 Governor (EMA-контроллер) — обязателен

**Объект управления:** strictness — квантильный порог отбора кандидатов на refine. Все тайлы выше порога проходят, расход плавает.

**Сигнал обратной связи:** EMA фактического cost/step (после warmup).

**Цель:** удерживать cost/step в коридоре вокруг target_per_step (не «в среднем по больнице», а на каждом шаге).

**Предохранители:**

* Δstrictness за шаг ≤ clamp (anti-oscillation).
* hard_cap_per_step = 3× target (safety fuse от взрыва расхода).
* Warmup: первые N шагов strictness не двигается, но расход идёт.
* Compliance — асимметричная: overbudget penalty > underbudget penalty (перерасход = SLA-проблема, недобор = упущенная возможность).

**Экспериментальный результат (Exp0.8v5):**

* Governor снижает StdCost вдвое (~5.15 → ~3.25), P95 с 11.0 до ~6.5, compliance penalty с ~3.2 до ~0.5 — во всех 4 сценах.
* PSNR чуть ниже (−0.24 дБ clean, −0.68 дБ shift): фиксированный порог «жрёт» лучшие тайлы залпом, governor размазывает ровнее. При равном total cost governor проигрывает по пиковому качеству, но выигрывает по предсказуемости расхода.
* **Вывод:** governor нужен не для качества, а для контроля бюджета. Без него бюджет — декларация.

## 5.2 Условие применимости governor

Governor работает только если диапазон strictness реально меняет число кандидатов.

Если при max strictness n_candidates ≥ hard_cap — контроллер упирается в потолок и бесполезен.

Это лечится параметризацией задачи (больше тайлов, другой таргет, другой диапазон), не «ещё одной магией».

## 5.3 Phase schedule (веса ρ по шагам/глубине) — отложен

Exp0.8v5 не показал выигрыша в PSNR при depth-dependent весах (resid→anomaly по шагам).

Гипотеза: при идеальном refine и надёжной ρ менять приоритеты по фазе — оверинжиниринг.

Держать как опциональный extension на будущее, когда появится реальное дерево с неидеальным refine, где фазовая смена сигналов станет физически оправданной.

# 6. Boundary-aware refinement (обязательное требование)

Exp0.2–Exp0.3 показали:

Refinement без overlap / feather / halo при тайловой подстановке гарантированно создаёт ложные HF-артефакты при разреженном бюджете.

Причина: жёсткая вставка refined-тайла → ступенька на границе → Лапласиан ловит как ложный HF-сигнал.

Чем умнее adaptive выбор тайлов, тем сильнее артефакт (adaptive концентрирует refine в зонах с высоким градиентом, где ступенька бьёт сильнее).

**Минимальное правило:**

* overlap ≥ 3–4 пикселя при tile_size=16.
* Blending (cosine feather) выполняется относительно coarse.

Halo ≥ 3 пикселя полностью устраняет HF-деградацию. Interior-only HF метрика подтвердила: проблема была в швах, не в выборе тайлов.

---

# 7. Probe-budget exploration (обязательное требование)

Стратегия «уточнять только найденные границы» структурно слепа.

Каждый шаг:

* 90–95% бюджета → exploitation.
* 5–10% бюджета → probe.

Приоритет probe: coarse residual / variance, неопределённость, давность проверки.

Если probe выявляет активность — домен активируется.

**Trade-off:** на стационарных сценах probe может снижать PSNR (бюджет тратится «впустую»), но повышает шанс поймать сдвиги/редкие области. Probe contribution (ablation) в Exp0.8 подтвердил: вклад probe в PSNR минимален на статике, но probe — страховка от структурной слепоты, а не оптимизация качества.

Overlap не заменяет exploration. Exploration не лечит швы. Оба механизма обязательны.

---

# 8. Границы

Граница — зона затухания информативности.

Halo применяется по обе стороны границы.

---

# 9. Стратегии построения карты

1. Общая карта плотности (переиспользуемая).
2. Смысл-специфическая карта (быстрая, но одноразовая).

---

# 10. Ключевые инварианты

1. Уточнение аддитивно (residual).
2. Halo обязателен.
3. Exploration обязателен.
4. Стоимость управляется governor'ом (EMA-контроллер) в рамках бюджета.
5. Система не создаёт искусственные швы.
6. Система не становится структурно слепой.
7. ρ определяет семантику карты; комбинирование через двухстадийный гейт.

---

# 11. Главный принцип Curiosity

Curiosity — система, которая:

1. Считает только там, где есть сигнал.
2. Делает это без создания искусственных швов (halo).
3. Не становится слепой к внутренней структуре (probe).
4. Управляет бюджетом осознанно (governor), а не декларативно.
5. Адаптирует функцию интересности к условиям (двухстадийный гейт), а не полагается на один датчик.

Формально: refinement must be boundary-aware, must include controlled exploration, and must be budget-governed.

# 12. Экспериментальный статус (Exp0.1–Exp0.8)

## Подтверждено

| Эксперимент | Вопрос | Результат |
|---|---|---|
| Exp0.1 | Работает ли adaptive refinement? | Да: adaptive > random по MSE/PSNR при любом coverage. |
| Exp0.2 | Устойчиво ли на реальных изображениях? | Да по MSE_rgb/PSNR (98–100% winrate). HF-деградация при малом бюджете → мотивация для halo. |
| Exp0.3 | Нужен ли halo? | Да: halo ≥ 3 px устраняет HF-артефакты. Interior-only HF подтвердил: проблема в швах. |
| Exp0.4 | Нужна ли комбинированная ρ? | При чистых данных: residual-only = оракул. Комбинация не нужна. |
| Exp0.5 | Ломается ли оракул? | Да: при шуме, blur, alias residual деградирует (corr 0.90 → 0.54). |
| Exp0.6 | Работает ли бинарный переключатель? | Частично: clean/blur → resid, noise → combo. Alias — пограничный случай. |
| Exp0.7/0.7b | Нужен ли soft gating? | Двухстадийный гейт: решает clean/blur (Δ ≈ 0), побеждает на noise/spatvar (+0.77–1.49 дБ). |
| Exp0.8 | Нужен ли schedule? | Governor (EMA): нужен для контроля бюджета (StdCost −50%, penalty −85%). Phase schedule: не нужен при текущих условиях. |

## Открытые вопросы (следующая иерархия проверок)

* Нужна ли автонастройка порогов гейта? (сейчас — ручные instability/FSR thresholds)
* Нужна ли сложная структура данных? (Morton / динамический список vs. fixed grid)
* Даёт ли дерево смысловую метрику? (кусты, LCA-расстояние как feature)
* Как себя ведёт система при неидеальном refine? (delta ≠ GT − coarse)
* Phase schedule при неидеальном refine — оживает ли?
* Как обеспечить «не сломать фичи»? Adaptive refinement по определению создаёт неоднородное представление: часть пространства уточнена, часть — нет. Если downstream-потребитель (модель, loss, следующий слой) ожидает согласованный вектор признаков, неоднородная глубина уточнения может разрушить взаимосвязи между признаками. Halo лечит локальные швы, но не гарантирует глобальную согласованность. Нужен механизм или инвариант, обеспечивающий, что представление на любом уровне «матрёшки» остаётся валидным входом для потребителя — а не только визуально гладким.
