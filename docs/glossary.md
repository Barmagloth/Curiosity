# Глоссарий Curiosity

Термины проекта, без которых документация превращается в абракадабру. Отсортированы по смысловым группам.

---

## Базовые понятия

**Adaptive refinement (адаптивное уточнение)** — ядро системы. Идея в том, что пространство не уточняется целиком, а только там, где функция информативности ρ показывает, что это оправдано. Противоположность — uniform refinement (уточнять всё подряд) или random selection (выбирать случайно).

**Coarse (грубый уровень)** — начальное приближение, с которого стартует система. Вся дальнейшая работа — уточнение поверх этого уровня. Формула: `output = coarse + delta`.

**Delta (дельта)** — аддитивная поправка к coarse-уровню. Инициализируется нулём, ограничена по энергии. При отключении текущего уровня уточнения система откатывается к предыдущему валидному состоянию.

**Refinement (уточнение)** — процесс: вычислить ρ → решить, нужен ли split → уточнить выбранные регионы → повторить. Каждая итерация увеличивает разрешение только в «интересных» областях.

**Split / Merge** — split делит регион на подрегионы (обычно 4, как в quadtree); merge — обратная операция, схлопывание детей обратно в родителя. Гистерезис (разные пороги для split и merge) защищает от дрожания.

## Пространство и его элементы

**Ячейка (cell)** — элементарная единица дискретизации пространства произвольной размерности. Обобщение: pixel — ячейка в 2D-изображении, voxel — ячейка в 3D-объёме. В Curiosity пространство X может быть любой природы (признаки, латент, активации), поэтому используется общий термин «ячейка». В экспериментах на изображениях ячейка = пиксель.

**Тайл (tile)** — прямоугольный блок ячеек фиксированного размера на определённом уровне дерева. Базовая единица, над которой работает система: тайлы выбираются для уточнения, кэшируются, хэшируются. tile_size=16 означает блок 16×16 ячеек.

**Пространство X** — абстрактное пространство состояний, с которым работает Curiosity. Может быть чем угодно: пиксели изображения, латентные представления, активации нейросети, любые данные. Система не привязана к конкретной предметной области.

## Функция информативности и сигналы

**ρ (rho, «ро»)** — функция информативности. Определяет, где уточнение имеет смысл. Семантика дерева целиком определяется ρ. Может включать несколько компонентов (residual, HF, variance и др.).

**Интересность (interestingness)** — неформальное название для ρ. «Интересный» регион — тот, где ρ высока и уточнение оправдано. Политика интересности определяет, как балансировать разные сигналы на разных уровнях глубины.

**Residual (остаток)** — ошибка текущей аппроксимации. Основной сигнал информативности. На чистых данных residual-only = идеальный (оракул), но при шуме/blur/alias деградирует (корреляция с оракулом падает с 0.90 до 0.54).

**HF energy (высокочастотная энергия)** — структурная энергия, вычисляемая через Лапласиан или градиент. Ловит резкие переходы, но даёт ложные срабатывания на швах между тайлами.

**Variance (дисперсия)** — локальная дисперсия или disagreement между источниками. Чувствительна к шуму (что плохо) и к реальной неопределённости (что хорошо).

**Окупаемость (payoff)** — ожидаемый выигрыш от уточнения минус стоимость. Split допускается только если gain > cost.

**Оракул (oracle)** — идеальный, недостижимый на практике сигнал (ground truth). Используется для оценки: насколько наш реальный сигнал коррелирует с оракулом. Oracle-информация запрещена в production-метриках — это только для валидации.

## Двухстадийный гейт

**Двухстадийный гейт (two-stage gate)** — каноническая архитектура комбинирования сигналов ρ. Решает проблему: residual хорош на чистых данных, но ломается при шуме; комбинация помогает при шуме, но вредит на чистых.

**Stage 1** — бинарная проверка «Residual здоров?» по метрикам instability и FSR. Если здоров — используется residual-only (нулевые потери на чистых данных). Если нет — переход к Stage 2.

**Stage 2** — utility-weighted комбинация. Каждый эксперт (resid, var, hf) получает вес: `U_i = median(gain) − λ·FSR − μ·instability`. Residual имеет минимальный гарантированный вес. Веса сглажены EMA с гистерезисом.

**Instability (нестабильность)** — метрика надёжности сигнала. Высокая instability означает, что residual «скачет» и ему нельзя доверять. Порог для переключения Stage 1 → Stage 2.

**FSR (False Signal Rate)** — доля ложных срабатываний. Высокий FSR означает, что сигнал часто указывает «здесь интересно» там, где на самом деле ничего нет.

## Границы и швы

**Halo (гало)** — зона перекрытия между тайлами. Обязательный компонент: без halo жёсткая вставка refined-тайла создаёт ступеньку на границе, которую Лапласиан ловит как ложный HF-сигнал. Минимальный размер: 3 ячейки. Реализуется как cosine feathering относительно coarse.

**Cosine feathering (косинусное сглаживание)** — метод плавного перехода между refined и coarse тайлами. Вес cosine-функцией плавно меняется от 1 (центр тайла) до 0 (граница), вместо резкого обрыва.

**Шов (seam)** — граница между тайлами с разным уровнем уточнения. Искусственные швы — артефакт, который система обязана не создавать. SeamScore измеряет их тяжесть.

**SeamScore** — метрика качества швов. Формула: `SeamScore = Jumpout / (Jumpin + eps)`. Jumpout — градиент наружу от границы, Jumpin — градиент внутрь. Чем ближе к 1, тем лучше (шов незаметен). Валидирована на 4 типах пространств: 2D скалярные сетки, векторные сетки, нерегулярные графы, иерархии деревьев. Production-ready.

**Edge strips (краевые полоски)** — узкие полоски ячеек вдоль границы тайла, по которым вычисляется SeamScore.

## Probe и exploration

**Probe (зонд, разведка)** — выделенная часть бюджета (5–10%) на исследование неизвестных регионов. Без probe система видит только то, что уже нашла (exploitation-only = структурная слепота).

**Exploration / Exploitation** — exploitation — уточнение уже найденных «интересных» областей; exploration — поиск новых. 90–95% бюджета на exploitation, 5–10% на exploration. Оба обязательны: exploration не лечит швы, halo не заменяет exploration.

**Структурная слепота** — состояние системы, когда она не видит внутреннюю структуру регионов, потому что никогда их не проверяет. Probe — страховка от этого.

## Budget Governor

**Governor (управляющий)** — EMA-контроллер, управляющий strictness для удержания расхода в бюджете. Без governor'а бюджет — декларация, а не ограничение.

**Strictness (строгость)** — квантильный порог отбора тайлов-кандидатов на refinement. Высокая strictness = только самые «интересные» тайлы проходят, расход ниже. Низкая = больше тайлов, расход выше. Governor двигает strictness чтобы cost/step оставался в коридоре.

**EMA (Exponential Moving Average)** — экспоненциально взвешенное скользящее среднее. Используется для сглаживания: весов в гейте, сигнала обратной связи в governor, и т.д. Реагирует на свежие данные сильнее, чем на старые.

**Compliance (соответствие бюджету)** — метрика того, насколько фактический расход соответствует целевому. Асимметричная: overbudget (перерасход) штрафуется сильнее, чем underbudget (недобор).

**Warmup (разогрев)** — начальные N шагов, во время которых governor не двигает strictness, но накапливает статистику расхода. Нужен для корректной калибровки.

**Hard cap** — жёсткий потолок расхода за шаг (= 3× target). Safety fuse от взрыва расхода.

## Дерево и хранение

**Дерево уточнения (refinement tree)** — журнал маршрутов решений split. Каждый путь root→leaf = последовательность решений. Не обязательно даёт «смысловую» метрику (это проверяется в P3).

**Куст (bush)** — множество путей в дереве, ведущих к одному смыслу. Формально: кластер leaf-путей с малым LCA-расстоянием.

**LCA (Lowest Common Ancestor)** — ближайший общий предок двух узлов в дереве. LCA-расстояние = мера похожести двух путей. Чем ближе LCA к корню, тем пути дальше друг от друга.

**Morton layout** — способ упаковки данных в памяти по Z-кривой (space-filling curve). Отвергнут: overhead на сортировку съедает выигрыш. Мёртв.

**Block-sparse layout** — разреженное хранение блоками. Отвергнут: expansion ratio делает его неэффективным. Мёртв.

**Compact layout** — компактное хранение только активных ячеек. Единственный выживший кандидат (vs. grid). Exp0.9b0 — kill/go проверка.

**Dirty-сигнатура** — 12-битная сигнатура (seam_risk + uncert + mass) для быстрого определения, изменился ли регион. Debounce (2 последовательных попадания) защищает от ложных срабатываний на шуме.

## Метрики

**MSE (Mean Squared Error)** — среднеквадратичная ошибка. Базовая метрика качества реконструкции.

**PSNR (Peak Signal-to-Noise Ratio)** — пиковое отношение сигнал/шум, в децибелах. Выводится из MSE. Выше = лучше. Разница в 0.5 дБ — заметна, 1+ дБ — существенна.

**Winrate** — доля сценариев, в которых метод A побеждает метод B. 98–100% = однозначное превосходство.

**StdCost** — стандартное отклонение расхода. Мера предсказуемости бюджета. Governor снижает StdCost на ~50%.

**P95** — 95-й перцентиль расхода. Мера worst-case. Governor снижает P95 с 11.0 до ~6.5.

## Методология

**Kill criteria** — условия, при которых эксперимент закрывается как неуспешный. Определяются до запуска. Двусторонние: speed + memory. Если не проходит — эксперимент мёртв, а не «допиливается».

**Cost-fair comparison** — сравнение, в котором вычислительные затраты учтены. Нельзя сравнивать метод с 10× overhead с baseline и говорить «мы лучше по PSNR» — нужно учесть стоимость.

**Observable-only** — в метриках используется только то, что система может наблюдать сама, без oracle-информации. Нельзя подсматривать ground truth при оценке.

**Holm-Bonferroni** — статистическая коррекция для множественных сравнений. Если проводишь N тестов, вероятность ложного срабатывания растёт. Holm-Bonferroni корректирует пороги значимости.

**Ablation** — удаление одного компонента системы и измерение, насколько ухудшился результат. Показывает вклад компонента.

## Иерархия приоритетов

**P0–P4** — уровни приоритета экспериментов:
- **P0**: GPU layout (фундамент, без него всё остальное — декоративное)
- **P1**: Компрессия дерева (dirty-сигнатуры, segment compression, anchors)
- **P2**: Автонастройка порогов гейта (instability/FSR)
- **P3**: Семантика дерева (LCA-расстояние, кусты, кластерность)
- **P4**: «Не сломать фичи» (downstream compatibility)

Порядок строгий: нельзя прыгать вперёд без закрытия зависимостей.

**Критический путь**: P0 → P1 → P3 → P4. P2 идёт параллельно с P1.

## Типы сценариев (используются в экспериментах)

**Clean** — чистые данные без искажений. Baseline-сценарий.

**Noise** — случайный шум, добавленный к данным. Residual деградирует.

**Blur** — размытие / расфокусировка. Residual справляется (Stage 1 остаётся).

**Alias** — артефакты частотного наложения (Найквист). Пограничный случай для гейта.

**Spatvar** — пространственная вариативность (spatial variance). Неравномерная сложность сцены.

**JPEG** — артефакты сжатия. Единственный сценарий с минусом для двухстадийного гейта (−0.21 дБ).

**Shift** — сдвиг / изменение сцены во времени. Тест на адаптивность.

## Замороженные концепции

**C (DAG + профили)** — маршрутизация уточнения по направленному ациклическому графу вместо дерева. Заморожена бессрочно. Входной контракт для размораживания: (1) минимум 2 несводимые цели, (2) конкретный downstream consumer, (3) наблюдаемый конфликт между целями.

**Matryoshka invariant** — требование, чтобы представление на любом уровне «матрёшки» (вложенного уточнения) было валидным входом для downstream-потребителя. Не только визуально гладким, а функционально корректным. Проверяется в P4.
